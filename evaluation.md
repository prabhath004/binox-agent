# Evaluation - Architecture Trade-offs

## Executive Summary

This project was intentionally built as a bounded research system rather than a general-answer chatbot. The key design goal was not to maximize answer length or model autonomy, but to produce explainable behavior under explicit memory, retrieval, and cost limits.

The system answers in-scope research questions over a curated corpus of AI developer tooling documents. It does so through a staged pipeline:

1. plan the research
2. retrieve evidence
3. compress evidence when memory pressure rises
4. replan if critical coverage gaps remain
5. synthesize a grounded report

The sections below document why these choices were made, what they buy us, and where they intentionally trade off completeness for cost control and reliability.

## 1. Why a Pipeline Instead of a Single Prompt

The pipeline decomposition (`plan -> retrieve -> compress -> replan -> synthesize`) was chosen over a single-shot prompt for three reasons.

### Modularity

Each stage has a narrow responsibility:

- planner: decompose the question
- retriever: fetch relevant corpus evidence
- compressor: reduce evidence size while preserving key facts
- replanner: add follow-up sub-questions only when evidence is insufficient
- synthesizer: produce the final answer

This makes the system easier to inspect and debug than a monolithic prompt where failures are opaque.

### Constraint enforcement

Per-step token limits, chunk limits, and replan limits are much easier to enforce between stages than inside a single large prompt. The pipeline structure makes limits first-class runtime behavior rather than documentation-only promises.

### Observability

The system exposes:

- sub-questions generated by the planner
- chunk counts and budget events
- memory/compression state
- limitations and source usage in the final output

That observability matters more in an evaluation setting than raw answer fluency.

### Trade-off accepted

The pipeline requires more LLM calls than a single prompt, so latency is higher. This is acceptable because the assignment rewards thoughtful constraint handling and architecture clarity more than raw speed.

## 2. Why Retrieval Instead of Passing the Whole Corpus

The corpus is small enough that it could be dumped into a large-context prompt. That was explicitly not the design goal.

### Why retrieval was still chosen

- the assignment is about operating under memory/token constraints
- retrieval architecture scales to much larger corpora
- bounded top-k retrieval makes cost behavior predictable
- smaller evidence windows improve focus during synthesis

### Why ChromaDB

Chroma was chosen because it is:

- local
- zero-config
- free to run
- sufficient for a bounded assessment corpus

### Trade-off accepted

Chroma's default local embedding model is practical, not state-of-the-art. If this system were taken further, the next retrieval improvement would be a reranking stage or stronger embedding model.

## 3. Memory Architecture Choice

The system uses a summarization-cascade memory strategy rather than a sliding context window.

### What this means

- raw retrieved chunks are stored first
- when accumulated evidence exceeds the context budget, the evidence is compressed into structured notes
- compressed summaries replace raw chunks for downstream synthesis

### Why this is useful

- preserves more useful information than naive truncation
- keeps prompts inside explicit context budgets
- lets the agent retain cross-question evidence across steps

### Why not a sliding window

A sliding window is simpler, but it silently drops earlier evidence. For research-style tasks, that is risky because it can erase important pricing or risk details discovered early in the run.

### Trade-off accepted

Compression is lossy. It preserves salient facts better than truncation, but it can still lose nuance and edge cases.

## 4. Constraint Model and Enforcement

The project enforces four explicit limits.

| Constraint | Default | Why it exists |
|---|---|---|
| Max context tokens per step | `800` | Prevent oversized prompts and force compression |
| Max retrieved chunks | `20` | Keep retrieval bounded and observable |
| Max cost per run | `$0.05` | Make run cost predictable |
| Max replans | `2` | Prevent unbounded loops |

### What happens when limits are hit

- chunk limit reached: lower-priority chunks are skipped
- context limit exceeded: evidence is compressed
- cost limit exceeded: the graph stops and synthesizes early
- replan limit reached: the pipeline stops expanding and proceeds to synthesis

### Hardening added after the initial prototype

The first version was structurally sound but too forgiving. The hardened version now also includes:

- strict input validation for invalid budgets or blank queries
- preserved plan state during hard cost cutoffs
- zero-evidence safeguards so the research path does not fake grounded answers
- regression tests for the failure modes most likely to hurt evaluator trust

## 5. Why FastAPI Owns Routing Instead of n8n

The prompt suggested `n8n/Dify for query routing + memory management`. The final implementation uses n8n, but not as the decision engine.

### Final split of responsibilities

**n8n**

- receives webhook requests
- performs light request validation/normalization
- proxies to FastAPI `/route`

**FastAPI**

- classifies queries
- decides between direct GPT and research pipeline
- handles zero-evidence fallback
- exposes health and API docs

**LangGraph**

- runs the actual research loop

### Why this was the better engineering choice

Putting routing logic in both Python and n8n made behavior harder to test and reason about. Centralizing routing in code gives:

- one source of truth
- easier regression testing
- fewer visual-workflow edge cases
- cleaner reproducibility for reviewers

### Trade-off accepted

This is slightly less literal to the "n8n for query routing + memory management" phrasing, but it is materially stronger from a correctness and maintainability perspective.

## 6. Reliability Over Optimism

One of the most important design choices in the final submission is that the agent now prefers to be visibly incomplete rather than confidently wrong.

### Examples

- If retrieval returns nothing relevant, the system returns a "no corpus evidence found" result instead of synthesizing unsupported claims.
- If `/route` sends an in-scope query to research but the corpus still returns nothing, the system can fall back to direct GPT with explicit limitations rather than pretending the corpus answered it.
- If cost budget is exceeded immediately after planning, the response still includes the sub-questions that were already generated.

This design choice matters because a budget-constrained research system should be auditable, not just fluent.

## 7. Business and Product Reasoning

The project is small, but the cost/benefit framing is realistic.

### Why route at all

Not every user question deserves the expensive research path. Routing off-topic or general-knowledge queries away from retrieval:

- saves cost
- reduces latency
- avoids irrelevant vector lookups
- keeps the research agent focused on the corpus it actually knows

### Why bounded memory matters to a client

Clients care less about architectural elegance than about predictable operational behavior. Explicit limits provide:

- predictable spend
- bounded latency
- easier debugging
- fewer failure modes during growth

### Why the corpus is curated

For an assessment, a curated corpus provides:

- reproducibility
- inspectability
- stable expected behavior

Web crawling would be broader, but much harder to reproduce and evaluate consistently.

## 8. Remaining Gaps

This is a strong prototype, not a finished production platform.

Main remaining limitations:

- no reranking layer on top of vector retrieval
- no quantitative benchmark suite for answer-quality scoring
- synchronous OpenAI client underneath the threadpooled FastAPI wrapper
- local n8n deployment is for development/demo use, not hardened deployment
- corpus scope is bounded and manually curated

## 9. Design Summary

| Decision | Choice | Why |
|---|---|---|
| Research orchestration | LangGraph | Native fit for shared state and conditional loops |
| Retrieval store | ChromaDB | Local, free, reproducible, simple |
| LLM | `gpt-4o-mini` | Cheap JSON-capable model for repeated structured calls |
| Memory strategy | Summarization cascade | Better than truncation for bounded research loops |
| Routing owner | FastAPI `/route` | Single source of truth, easier to test |
| Workflow tool | n8n | External integration layer without duplicating core logic |
| Corpus | Curated markdown files | Reproducible and inspectable for assessment purposes |


That is the right shape for a deep-research assessment built around constraints rather than unconstrained generation.
